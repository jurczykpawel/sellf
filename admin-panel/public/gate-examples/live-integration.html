Element gating — deploy reference

The exact HTML to put on your own page. To see the behaviour interactively first, open the interactive demo →

Before you start

  1. Add your page’s domain to the product’s allowed origins (Sellf admin → Settings → Embed). Without it the gate redirect is rejected with 400.
  2. Generate the snippet from Products → ⋯ → “Generate gating snippet” (it fills in your Sellf origin and product slugs for you), or adapt the template below.
  3. Replace YOUR-SELLF.example with your Sellf origin and the slugs with your product slugs.

Paste in <head>

<script src="https://YOUR-SELLF.example/api/loginwall/gate.js?products=pro-kit,vip-masterclass"></script>
<script>!window._SF_GATE_XXXX && (location.href =
  "https://YOUR-SELLF.example/loginwall/gate?products=pro-kit,vip-masterclass&redirect=" +
  encodeURIComponent(location.href));</script>
<noscript><meta http-equiv="refresh"
  content="0;url=https://YOUR-SELLF.example/loginwall/gate?products=pro-kit,vip-masterclass"></noscript>
The generated snippet uses a real _SF_GATE_<hash> flag name. On its first visit the page bounces once through Sellf and comes back with a signed token in the URL fragment; the runtime reads it, resolves every block, and strips the token from the URL.

Mark up the blocks in <body>

Each gated block lists the slug and up to three branches. Anything outside a data-sellf-product block stays public.

<!-- content that differs by visitor state -->
<div data-sellf-product="pro-kit">
  <div data-has-access>   …content for buyers…            </div>
  <div data-no-access>    …buy CTA for signed-in non-buyers… </div>
  <div data-no-session>   …log-in CTA for guests…         </div>
</div>

<!-- a button only owners can use (in-browser work, best-effort) -->
<button data-sellf-feature="pro-kit">Render captions</button>
Branches are optional. Provide only the states you care about. If a branch is missing for the current state, the runtime simply shows nothing for it.
⚠️ Display gating is best-effort, not protection. All branch HTML — including data-has-access — ships in your page source and is only hidden/removed in the browser. A determined visitor can read it from the page source or hand-edit the token. Use display gating for UX (show the right CTA), never to protect secret content. Anything that must stay private or costs money has to be gated server-side via SellfGate.verify() (next section) — or simply not shipped to the browser at all.

Gate a real backend action

In-browser locks are best-effort (a visitor can edit their own page). For anything that runs on a server or costs money, confirm access first — the server re-checks live ownership of the signed token, so it can’t be faked or replayed after access is revoked.

document.querySelector('#render-b').addEventListener('click', async () => {
  if (await SellfGate.verify('pro-kit')) {
    startServerRender();      // Sellf confirmed ownership server-side
  } else {
    location.href = '/p/pro-kit';   // send them to buy
  }
});

Test it

  1. Guest: open the page in a private window → you should see the data-no-session branches.
  2. Signed in, no purchase: sign in to Sellf, reload → data-no-access branches; features locked.
  3. Owner: with active access → data-has-access branches; features unlocked; the URL has no _sf_token after load.
